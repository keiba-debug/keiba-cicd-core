# フローとストック × KeibaCICD
> 細谷功『フローとストック』(KADOKAWA, 2024) をもとにした、プロジェクト思考の整理

---

## 1. なぜ今、この思考法なのか

KeibaCICDは「Continuous Integration & Continuous Delivery」を名に冠するプロジェクトです。  
これはエンジニアリングの継続的改善サイクルを競馬分析に持ち込む、という思想の宣言でもあります。

「フローとストック」の思考法は、まさにそのサイクルを言語化したフレームワークです。  
**私たちが何を回し、何を積み上げ、そして何を手放すべきか**を整理するために、この概念を共有します。

---

## 2. CAFSマトリックスとは

細谷功氏が提唱する「CAFSマトリックス」は、以下の2軸を組み合わせた4象限の分析フレームワークです。

- 横軸：**具体（Concrete）↔ 抽象（Abstract）**
- 縦軸：**フロー（Flow）↔ ストック（Stock）**

```
                    フロー (Flow)          ストック (Stock)
                ┌─────────────────────┬─────────────────────┐
  抽象          │  ② フローとしての抽象  │  ③ ストックとしての抽象 │
  Abstract      │  仮説・思考・パターン認識│  体系化された知識・知恵  │
                ├─────────────────────┼─────────────────────┤
  具体          │  ① フローとしての具体  │  ④ ストックとしての具体  │
  Concrete      │  生データ・日々の出来事  │  スキル・習慣・型        │
                └─────────────────────┴─────────────────────┘
```

### 4象限の定義

| 象限 | 名称 | 内容 |
|------|------|------|
| ① | フローとしての具体 | 五感で得られる未整理の一次情報。日々の経験・出来事・生データ |
| ② | フローとしての抽象 | 具体から法則性やパターンを見出した段階。仮説・アナロジー |
| ③ | ストックとしての抽象 | 体系化された知識・理論。解釈や意味付けが加わった「知恵」 |
| ④ | ストックとしての具体 | 実践で体得したスキル・習慣。無意識に遂行できる「型」や「勘所」 |

### サイクルの流れ

```
① 混沌（生データ）
      ↓  パターンを見出す
② 仮説・思考
      ↓  体系化・言語化する
③ 知識・モデル
      ↓  具体事象に当てはめる
④ スキル・型
      ↓  陳腐化・環境変化（アンラーン）
① 混沌（生データ）へ戻る
```

**重要なのは④→①の遷移**。過去の成功モデルを手放し、再びフローの状態に戻れるかどうかが、成長と停滞の分岐点です。

## 3. KeibaCICDへの適用

### 3-1. プロジェクトの各要素をマッピングする

| 象限 | KeibaCICDでの対応要素 |
|------|----------------------|
| ① フロー×具体 | 毎週のレース結果、調教タイム、リアルタイムオッズ変動、JRA-VANの生データ |
| ② フロー×抽象 | 「この馬は道悪が得意では？」「前走の負けは展開負け」などの仮説。NLPによるコメント解析 |
| ③ ストック×抽象 | LightGBMモデル、Isotonic Regressionによる確率キャリブレーション、調教師プロファイル、コース特性DB |
| ④ ストック×具体 | 期待値計算ロジック、週次バッチ処理スクリプト、Kelly基準による資金配分の型 |

### 3-2. CI/CDとCAFSサイクルの対応

KeibaCICDというプロジェクト名自体が、このサイクルの自動化を意図しています。

```
Continuous Integration  ＝  ①②のフロー処理
                             生データの収集・仮説の継続的な更新

Continuous Delivery     ＝  ③④のストック更新と適用
                             モデルの再学習・推奨馬の出力
```

**週次サイクルのイメージ：**

```
月〜木：生データ収集（①）
  ↓
金：前週の予測ミスから仮説を立てる（②）
  ↓
土朝：モデル更新・EV計算（③）
  ↓
土〜日：当日レースに適用・ベット判断（④）
  ↓
月：レース結果を①に戻す（アンラーン）
```

## 4. 最も重要な問い：④→①ができているか

### ストックが「重荷」になるサイン

競馬予測の文脈で、以下を感じたら**アンラーンのタイミング**です。

- 「このモデルで去年は当たったから」という過信
- バックテストに過学習したモデルを使い続けること
- コース改修・調教師の引退・騎手トレンドの変化を無視していること
- EV計算のロジックを疑わずに走らせること

> 「人生は積み重ねだと誰でも思っているようだ。ぼくは逆に、積みへらすべきだと思う。」  
> — 岡本太郎『自分の中に毒を持て』

競馬の環境は絶えず変化します。**③で固めた「知識」が半年後には陳腐化している**可能性を常に意識することが、長期プラス収支への道です。

### イノベーションはフロー型の発想から

「カイゼン」はストック型——既存手法の最適化です。  
一方、**真のイノベーション**はフロー型——今ない変数を見つけ出す活動です。

KeibaCICDが単なる「便利ツール」に留まらず、**継続的に市場の歪みを発見するプラットフォーム**であり続けるためには、③④に安住しない姿勢が不可欠です。

## 5. プロジェクト全体への示唆

### フロー型で動くべきこと
- 仮説の立案と検証（週次）
- 新特徴量・新データソースの探索
- モデルのドリフト検知
- コミュニティでの知見共有・Issue議論

### ストック型で管理すべきこと
- 確立されたMLパイプライン
- 調教師・コース・騎手プロファイルDB
- Kelly基準などのリスク管理ルール
- CI/CD自動化スクリプト

### 定期的にアンラーンすべきこと
- 特定コンペティションに過適合したモデルウェイト
- 「このオッズ帯は買わない」などの固定ルール
- 過去の成功パターンへの執着

---

## 6. まとめ

| | フロー型 | ストック型 |
|--|---------|----------|
| 強み | 変化への適応、新発見 | 効率化、再現性、安定 |
| 弱み | 知識が蓄積されない | 陳腐化リスク、固定化 |
| KeibaCICDでの役割 | 仮説生成・探索フェーズ | モデル運用・自動化フェーズ |

KeibaCICDが長期的に機能し続けるためには、**フロー（探索・更新）とストック（蓄積・運用）のバランス**を意識的に管理することが重要です。

そして何より、**④→①の遷移——過去の成功を手放し、再び混沌に飛び込む勇気**——を仕組みとして組み込んでいくことが、このプロジェクトの本質的な強さになると考えています。
