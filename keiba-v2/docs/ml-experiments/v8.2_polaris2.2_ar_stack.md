# v8.2 / polaris 2.2 — AR→W スタッキング (失敗実験 / archive only)

> ⚠️ **結論: 本番投入不可**
> AR1=W1 の一致率は改善したが、ROI が全戦略で悪化(W単勝 -6.4pt、W複勝 -47pt、ValueBet機能停止)。
> 2026-04-26 に live を polaris-2.1 にロールバック、2.2 は archive 保管のみ。

## 概要

AR(着差回帰)モデルの ability_score を W(勝利分類)モデルの入力特徴量として階層化する
スタッキング構成。Stars系AR(素の能力)とNebula系想定の市場・騎手特徴を W が統合する形に
変えることで、AR1≠W1 が34.8%発生する問題を構造的に解消する。

## 動機

polaris 2.1 (Session 117時点) の課題:
- AR1とW1が一致しないレース: **365/1050 = 34.8%**
- うち84%は Top3内の入れ替え、5.4%が「W>3位の激しい乖離」
- 原因:
  - 損失関数違い (W: binary logistic, AR: Huber regression)
  - **Optunaで選定された特徴量が乖離**:
    - W のみ: `jockey_*` 4本(騎手系・市場相関高)
    - AR のみ: `speed_idx_*` 5本 + `comment_*` 9本 (素の能力指標)

W は騎手込み、AR は素能力 — 視点が違うのは設計上の意図だが、ユーザー視点では
「結局どっちを買えばいいのか」が分かりづらい。階層化することで:
- AR は引き続き「素の能力」を出力
- W は「AR能力 × 市場・騎手」を統合した実用判断

## 変更内容

### 1. CLIフラグ `--ar-stack` 追加 (ml/experiment.py)

```bash
python -m ml.experiment --use-optuna --ar-stack --version 2.2
```

### 2. 学習順序の変更: P → AR → W

旧: P → W → AR (独立3モデル)
新: P → AR(K-fold OOF + full) → W(stacking with `ar_ability_score`)

### 3. K-fold OOF (リーク防止)

- `df_train` は 5-fold OOF で `ar_ability_score` を生成
- `df_val`/`df_test` は full-train AR モデルで予測 (out-of-sample)
- target_margin が NaN な行は full モデル予測で穴埋め

### 4. predict.py / backtest_bet_engine.py の推論順序更新

- AR を先に予測 → `ar_ability_score = -pred_margin_ar` を features 辞書に注入
- `features_per_model['w']` に `ar_ability_score` が含まれる場合のみ動作
- 旧モデル (polaris 2.1以前) との後方互換性を維持

### 5. meta.json への記録

- `ar_stack: true` フラグ追加
- `features_per_model['w']` に `ar_ability_score` が自動的に含まれる

backtest_bet_engine.py を 2.1/2.2 両方で実行、test set 2,934レースで実収支比較。

### Pモデル (影響軽微 〜 微改善)

| 指標 | polaris 2.1 | polaris 2.2 | 差分 |
|------|-------------|-------------|------|
| P Top1 単勝 ROI | 91.5% | 94.1% | +2.6pt |

| 指標 | polaris 2.1 | polaris 2.2 | 差分 |
|------|-------------|-------------|------|
| W Top1 勝率 | 21.4% | **29.6%** | +8.2pt (一見改善) |
| **W Top1 単勝 ROI** | **86.6%** | **80.2%** | **-6.4pt** |

**何が起きたか:**
- W に `ar_ability_score` を入れた結果、W の予想が「素能力 = 人気馬」と高相関になった
- 当てる頻度は上がる (+8.2pt) が、当てた馬の配当が暴落 (平均人気 3.29→1.64)
- 結果、**単勝・複勝・ValueBet 全て ROI 悪化**
- **ValueBet戦略 (穴狙い) が機能停止** — W が odds とほぼ同じ序列になったから

## 学び (本質)

### ❌ AR1≠W1 の不一致は「問題」ではなく「収益源」だった

当初の問題提起:「AR1とW1が一致しないことが多いのが気になる」(34%乖離)
実装の意図:「AR能力とW実戦予想を階層化して一致を促す」
実際の結果:**乖離していた馬こそが ValueBet 候補で、収益機会の中核だった。**

- AR と W が違う序列を出すのは設計欠陥ではなく、**Stars系 (素能力) と Nebula系 (実戦) の視点違いを活かした構造**
- W に AR を入れた瞬間に、W は AR と同じ「人気馬選好」に収束
- この人気馬選好は的中率を上げるが、回収率を下げる (オッズが低い馬ばかり選ぶため)

### ✅ 達成された(無意味な)指標

- AR1≠W1 9.7pt減 (32.3→22.6%) — **そもそも減らすべきでなかった**
- ar_ability_score W#1 重要度 (1.42M, 4.5倍) — **W が AR の劣化コピーになった証拠**
- W Top1 勝率 +8.2pt — **配当悪化を考慮すると意味なし**

### 設計原則の再確認

> 「異なる視点で予測するモデルを階層化すると、強い視点が弱い視点を吸収して、結果的に視点の多様性が失われる」

Stars系 (polaris/sirius/...) と Nebula系 (eclipse/tide/...) を別系統として並列運用するのが正しい設計。
スタッキングはモデル間の独立性を破壊するため、**収益源としての「視点の違い」を保ちたい場合は使うべきでない**。

1. ✅ live を polaris-2.1 にロールバック (2026-04-26)
   - `data3/ml/models/polaris/live/` に versions/vpolaris-2.1/ をコピー
   - `data3/ml/models/polaris/archive/v2.2/` に 2.2 を退避
   - `model_registry.json` の active_version を `polaris-2.1` に変更
2. `--ar-stack` フラグはコードに残置 (将来の参考実装として、ただしREADME相当に「収益悪化のため非推奨」と注記)
3. 次の改善方向: **B案 (target_margin の不利補正)** へ切り替え
   - AR の精度を「単独で」上げる方向。Stars系の本来の役割を強化
   - 不利補正済みmargin (`mae_furi_adj` 等) を target に使う

## 備考

- AR本体の精度 (MAE/Corr) は変わらない (AR学習プロセスは同じ)
- `ar_ability_score` は `features_value` のunionに自動追加され、
  predict.py側でAR予測後に注入されるため、入力特徴量NaN警告からは除外
- 失敗実験だが、「乖離 = 収益源」という重要な発見を残した実験
