# v5.7 血統カテゴリ特徴量実験

> Date: 2026-02-25 (Session 50)
> Status: **NG** - AUC悪化、コメントアウトして無効化

## 概要

父（sire）と母父（broodmare sire）の繁殖登録番号（HansyokuNum）をLabelEncodingし、
LightGBMの`categorical_feature`として直接投入する実験。
Zenn記事「競馬予想で始める機械学習」のアプローチを参考にした。

## 方法

1. UM_DATA（馬マスタ）から父/母父のHansyokuNumを抽出
2. pedigree_index.json を構築（211,787馬、58,512種sire、34,913種BMS）
3. train+val+test全体でLabelEncoding（sire→整数、bms→整数）
4. LightGBMの`categorical_feature`パラメータで指定
5. 特徴量名: `sire_encoded`, `bms_encoded`

## 分析

### 高カーディナリティ過学習問題

- **sire: 14,209種類、bms: 9,590種類**のカテゴリは過学習を引き起こした
- Model A (odds含む) が**iter=16で早期停止**。通常247回なのに93%削減
- Feature Importanceは高い（Model B 2位/3位）→ **情報量はあるが暗記に走る**
- 各sireの出走回数が平均13回程度では、カテゴリ分割が過度に特化

### 書籍との違い

Zenn記事では血統LabelEncodingが有効だった理由（推測）:
- より多い学習データ（longer history window）
- netkeiba.comのcategory体系が異なる可能性
- 62列（5世代）を同時投入 → 相互作用で汎化した可能性

## 代替アプローチ（未実装）

1. **集計特徴量**: sire_win_rate, sire_top3_rate, bms_win_rate 等
   - ベイズ平滑化で少数出走sireも安全に扱える
   - 過去走のrate系特徴量と同じパターン
2. **Target Encoding**: sire_idごとのtarget平均（smoothed）を数値投入
   - カテゴリではなく連続値 → 過学習しにくい
3. **頻度フィルタ**: N回以上出走のsire/bmsのみ残し、稀少IDを「その他」にまとめる
4. **min_data_in_bin増加**: LightGBMパラメータで最低サンプル数を制限

## 結論

**高カーディナリティID直接投入は失敗**。情報量はあるが、14K/10Kカテゴリは
LightGBMのcategorical_featureでは汎化しない。次回は集計特徴量アプローチで再挑戦する。
