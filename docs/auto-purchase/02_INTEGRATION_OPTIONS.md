# 02. 購入実行の統合オプション比較

> WEB調査実施: 2026-05-22。法的助言ではない。

## 1. 評価軸

| 軸 | 説明 |
|----|------|
| 実装コスト | KeibaCICD からの追加開発量 |
| 運用安定性 | IPAT仕様変更・障害時の耐性 |
| 完全自動度 | 人手の最終投票が要るか |
| 既存資産との親和性 | FF CSV / TARGET / Windows |
| 監査性 | 成否・金額・時刻の取得しやすさ |
| コスト | ライセンス・API利用料 |
| リスク | ToS・非公式・誤投票 |

## 2. オプション一覧

### Option A: TARGET IPAT連動（推奨・第一候補）

**概要**: 現行 FF CSV → TARGET 買い目取り込み → TARGET の「IPAT投票」で買い目自動入力 → **人間がIPAT画面で最終投票**。

**公式根拠**（JRA-VAN TARGET FAQ）:

- [IPAT連動機能設定](https://targetfaq.jra-van.jp/faq/detail?id=473)
- IPATは非公開仕様を独自解析。予告なく利用不可・誤入力の可能性
- 投票確認前に買い目・金額の目視確認が必須
- 複数レース一括時、一部失敗しても他は通るが、**成否がTARGETに返らない**場合あり → IPAT側で投票結果確認が必須

| 項目 | 評価 |
|------|------|
| 実装コスト | **低**（KeibaCICDはFF自動化+UIのみ。TARGET設定は運用） |
| 安定性 | 中（TARGET側が追随。KeibaCICDは影響小） |
| 完全自動度 | **50–70%**（最終投票は人手） |
| 親和性 | **最高**（既存 `target-pd-writer.ts`） |
| 監査性 | 低–中（成否APIなし。人手確認+IPAT履歴で補完） |

**KeibaCICD でやること**

1. 締切N分前に FF CSV 自動生成（Orchestrator）
2. TARGET 取込を促す UI 状態 + チェックリスト
3. 人間が「IPAT投票完了」を Web でチェック → ledger 更新
4. 将来: TARGET 自動取込が可能ならファイルウォッチ検討（非公式）

---

### Option B: JRA-IPAT-API（team-nave 等）

**概要**: [JRA-IPAT インターネット投票API](https://www.team-nave.com/system/jp/products/ipatapi/) 経由で HTTPS で投票。単勝・複勝・馬連・ワイド・三連・WIN5、入出金・残高・投票照会。

| 項目 | 評価 |
|------|------|
| 実装コスト | 中–高（新クライアント、認証、エラーハンドリング） |
| 安定性 | **高**（仕様公開前提） |
| 完全自動度 | **80–95%** |
| 親和性 | 中（FF CSV不要。別パス） |
| 監査性 | **高**（投票照会API） |
| コスト | **有償**（要見積・利用制限確認） |

**検討タスク**

- 利用制限・料金の問い合わせ
- PoC: 単勝100円×1レース
- 本番は Themis 出力 → API リクエストにマッピング層

---

### Option C: 即PAT / IpatHelper DLL

**概要**: [ipathelper_dll](https://github.com/Mikimini9627/ipathelper_dll) 等、Windows DLL で即PATログイン・購入。

| 項目 | 評価 |
|------|------|
| 実装コスト | 中（Python ctypes / 小さなC#ラッパー） |
| 安定性 | 低–中（非公式、JRA/即PAT更新で破綻リスク） |
| 完全自動度 | 高 |
| 監査性 | 中（`GetPurchaseData` 等あり） |

松風アーキテクチャでは L5 に **PAT/即PAT** が明示されているが、**メンテと障害時の責任分界**を文書化してから採用。

---

### Option D: ブラウザ自動化（Playwright / Selenium）

**概要**: IPAT Web UI を操作（コミュニティ事例あり）。

| 項目 | 評価 |
|------|------|
| 実装コスト | 中 |
| 安定性 | **低**（UI変更で即死） |
| 完全自動度 | 設定次第（`auto=False` が一般的） |
| リスク | ToS・CAPTCHA・セキュリティソフト |

**推奨**: 本番自動購入には**非推奨**。PoC・個人実験のみ。

---

### Option E: 第三者の「全自動」競馬ソフト

例: 諭吉くん等 — I-PATユーザ情報登録、発走2分前オッズDL→計算→投票。

| 項目 | 評価 |
|------|------|
| 実装コスト | 低（KeibaCICD外） |
| 統合 | **困難**（Themis・VB Floor と二重管理） |
| 推奨 | KeibaCICD の判断ロジックと分離されるため**非推奨** |

## 4. 推奨戦略（ハイブリッド）

```
短期（1–2スプリント）: Option A + 見える化UI + purchase_ledger
中期（1–2ヶ月）:       web-roadmap §13 JIT Phase 1–2 完了
中長期（判断後）:      Option B の PoC（並行調査のみ、本番切替は実績後）
```

**理由**

1. FF CSV と ExecuteTab の投資を無駄にしない
2. TARGET FAQ が明示する「最終投票は人手」を公式に受け入れ、**半自動でリスクを抑える**
3. 成否が取れない弱点は **ledger + 人間確認ステップ + IPAT Web での事後照合** で補う
4. 完全自動が必要になった時点で B を評価（税務ログも取りやすい）

## 5. 認証情報の扱い（全オプション共通）

| 原則 | 実装案 |
|------|--------|
| リポジトリに載せない | `.env.local` / Windows Credential Manager |
| Web UI に表示しない | サーバー側のみ。管理画面はマスク |
| ローテーション | 暗証番号変更時の手順書 |
| 監査 | 「誰がいつ承認したか」は ledger に記録（パスワードは記録しない） |

TARGET IPAT連動の認証は **TARGET 環境設定に保存**（KeibaCICD 外）。API/DLL ルートを取る場合のみ KeibaCICD 側 vault が必要。

## 6. 締切タイミング（運用設計用）

| イベント | 目安 | Orchestrator |
|----------|------|--------------|
| 発走 | T | — |
| 投票締切 | 一般に T-2分〜T-1分（券種・場により異なる） | **T-10分**: EV再計算、**T-7分**: FF出力、**T-5分**: 人間IPAT確認デッドライン |
| vb_refresh | 5分間隔 | 最後の refresh が T-10 より後なら警告 |

※ 正確な締切は `race_shosai` または keibabook の発走時刻から算出。実装時は **安全マージン +2分** を推奨。

## 7. 失敗モードとリトライ（松風教訓反映）

| 失敗 | 検知 | 対応 |
|------|------|------|
| FF書込み失敗 | ファイルなし / API 500 | 1回リトライ、失敗で `FAILED` + アラート |
| TARGET 未取込 | 人間タイムアウト | `STALE`、次レースへ予算戻し |
| IPAT 通信エラー | TARGET FAQ 上の既知リスク | 購入停止、手動投票 SOP |
| 一部レースのみ失敗 | 一括投票 | IPAT「直前の投票結果」確認必須 |
| 二重投票 | 同一 race_id + ledger | 冪等キー `race_id+preset+hash(bets)` |
