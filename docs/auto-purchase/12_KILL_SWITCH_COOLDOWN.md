ふくだ君。シズネは初稿で「自動停止 → 段階クールダウン → 自動復帰」のモデルを提案しました。**この前提は今回ふくだ君判断で却下されました。** v2.0 でモデル自体を書き直します。

ふくだ君の根本的な意思は **「自動購入の開始自体を、毎朝、人が必ず実行する」** です。翌朝09:00 になったらシステムが勝手に再開する設計を作ると、ふくだ君は「再開したことに気付かないまま」資金が動くリスクを負います。Themis 原則は「確率 × オッズ × bankroll」だけで購入判断を曲げないことですが、**「今日も自動運転していいか」の判断レイヤだけは人に残す** という整理です。これは確率・オッズ・bankroll の3変数より一つ上の階層、「**今日この自動システムを動かす意思があるか**」を毎朝確認する仕組みになります。

結果として、kill switch の役割は劇的にシンプルになります。

- **当日中**: DD 超過 / bankroll 枯渇 / 突合 🔴 検出 → 即座に当日の自動購入を停止
- **翌日**: 何も自動復帰しない。前日の状況サマリを見てふくだ君が「開始する」「今日は休む」を選ぶ

「停止/復帰」モデルから「**毎日オプトイン**」モデルへの転換です。

## 1. 仕様表

### 1.1 当日中の停止トリガー（コア）

| # | 発動経路 | 発動条件 | mode | 当日中の挙動 |
|---|---|---|---|---|
| 1 | **手動** | ふくだ君が緊急停止ボタンを押した | `stopped` | 当日中は再開不可。翌朝のオプトイン UI で改めて判断 |
| 2 | **自動・DD** | DD -20% / -30% / -40% のいずれかに到達 | `stopped` | 同上。DD レベルは UI に表示するがクールダウン時間の段階分けは無し |
| 3 | **自動・bankroll** | bankroll < 初期 10% | `stopped` | 同上 |
| 4 | **自動・突合 🔴** | 当日中に [13_AUTO_RECONCILIATION.md](./13_AUTO_RECONCILIATION.md) §2.2 の 🔴 高 が **1 件以上** 検出 | `stopped` | 同上 |

> **v2.0 設計判断**: クールダウン時間（黄30分・橙60分・赤翌朝09:00）の段階分けは廃止。理由 — 翌日に自動復帰させない以上、当日中はどのレベルでも「**当日終了まで停止**」の一択で十分。段階分けは「自動復帰のタイミング差」を表す概念だったので、自動復帰廃止と同時に概念ごと不要になる。

### 1.2 「当日終了」の定義（v2.1 動的決定）

| 案 | 採否 | 理由 |
|---|---|---|
| **「最終レース終了 + 30分」を当日終了時刻として動的決定** | ✅ **採用（v2.1）** | 実態に即している。深夜帯まで `emergency_stop=true` のフラグを引きずらず、翌朝の動作（00:00 を待たずに reset 可能）に近付けられる。「最終レースが終わって、的中払戻が IPAT に反映され、30 分置けば事後操作の整理時間も確保できる」という運用観に沿う |
| 当日 23:59:59 JST 固定 | ❌ v2.0 → v2.1 で廃止 | 実態と乖離（最終レースが 16:30 終了でも 23:59 までフラグが残るのは無駄）。ただし「最終レース時刻が取得できない/取消・順延した日」のフォールバックとして残す（§8-11 参照） |
| 24 時間経過後に解除 | ❌ | 「当日の終了」と「24時間後」がズレるので人間直感に反する |
| 翌朝 09:00 自動復帰 | ❌ **v1 で提案・v2 で廃止** | そもそも自動復帰自体を廃止する方針なので段階分けの意味が消える |

**動的決定の実装概要**:

```typescript
// 擬似コード
function computeEodAt(date: string): string {
  const schedule = loadRaceSchedule(date);  // race_*.json or DB
  const lastPostTime = max(schedule.map(r => r.post_time));  // 最終レース発走時刻 JST
  if (!lastPostTime) {
    // フォールバック: スケジュール取得失敗 or 当日無開催 → 23:59:59 JST
    return `${date}T23:59:59+09:00`;
  }
  // 発走時刻 + レース所要時間(平均3分) + 30分バッファ
  return addMinutes(lastPostTime, 33).toISOString();
}
```

実装上の注意点は §8-11 にまとめる。

**重要**: `eod_at` を過ぎても、翌朝の自動購入は **ふくだ君が明示的に開始ボタンを押すまで** 一切動きません（§4 参照）。「当日終了」は単に「`emergency_stop` フラグの自動 clear と mode→`manual` リセット」を意味するだけで、自動運転の再開は意味しません。

### 1.3 廃止された v1 仕様（記録のため明記）

以下は v1 で提案し、v2 で廃止された仕様。実装者は迷ったらこの表を見てください。

| v1 仕様 | v2 での扱い | 廃止理由 |
|---|---|---|
| 黄 30分 / 橙 60分 / 赤 翌朝09:00 の段階クールダウン | ❌ 廃止 | 自動復帰しないので段階分け不要 |
| 翌朝09:00 自動復帰 | ❌ 廃止 | 毎朝オプトイン UI で人が判断する |
| 突合連動 N=2（橙60分）/ N=3（赤・翌朝09:00） | ❌ 廃止 | 連続日数を機械が追う前提は「人が毎朝判断する」モデルでは不要。毎朝ふくだ君が前日サマリを見て自分で判断する |
| `consecutiveHighDays` カウンタの永続化 | ❌ 廃止 | 連続日数概念ごと廃止 |
| 解除時の二重確認ダイアログ + 理由必須 | ❌ 廃止 | 当日中の解除自体が無いので不要 |
| `kill_switch.unlock_history` | ❌ 廃止 | 当日中の解除自体が無いので履歴不要。代わりに翌朝オプトイン履歴を §4 で記録 |

## 2. 判定ロジック

### 2.1 発動判定（`/api/bankroll/check` 拡張）

[05_RISK_COMPLIANCE.md](./05_RISK_COMPLIANCE.md) §3 の DD 4層防御と、本書の当日停止を統合。

```typescript
// 擬似コード — /api/bankroll/check 内
interface KillSwitchEval {
  shouldStop: boolean;
  triggerReason: "manual" | "dd_yellow" | "dd_orange" | "dd_red"
               | "bankroll_critical" | "recon_high" | null;
  triggerDetail?: string;  // "dd_-32" / "recon_duplicate_in_ipat" 等
  stoppedUntilEod: boolean;  // 常に true（当日中固定）
}

function evaluateKillSwitch(
  state: BankrollState,
  recon?: ReconciliationState  // 当日突合の状態。null なら判定スキップ
): KillSwitchEval {
  const ddPct = (state.current - state.peak) / state.peak * 100;
  const bankrollPct = state.current / state.initial * 100;

  // 突合 🔴 が1件でも検出 → 即停止
  if (recon?.highSeverityCount && recon.highSeverityCount >= 1) {
    return _stopUntilEod("recon_high", recon.highSeverityReasons.join(","));
  }

  // bankroll 危機
  if (bankrollPct < 10) return _stopUntilEod("bankroll_critical", `bankroll_${bankrollPct.toFixed(1)}pct`);

  // DD（レベル表示のためトリガ理由は分けるが、挙動は同一）
  if (ddPct <= -40) return _stopUntilEod("dd_red", `dd_${ddPct.toFixed(1)}`);
  if (ddPct <= -30) return _stopUntilEod("dd_orange", `dd_${ddPct.toFixed(1)}`);
  if (ddPct <= -20) return _stopUntilEod("dd_yellow", `dd_${ddPct.toFixed(1)}`);

  return { shouldStop: false, triggerReason: null, stoppedUntilEod: false };
}

function _stopUntilEod(reason: KillSwitchEval["triggerReason"], detail: string): KillSwitchEval {
  return {
    shouldStop: true,
    triggerReason: reason,
    triggerDetail: detail,
    stoppedUntilEod: true,
  };
}
```

> **判定順の理由**: 突合 🔴 は「お金が消える経路の最も確度高いシグナル」なので最優先。次に bankroll、最後に DD レベル。`recon_high` は DD と独立に発火する（DD が浅くても突合異常で停止する）。

### 2.2 「連続日数」を判定しない理由

v1 では「`ledger_only` が開催日連続 2 日で橙、3 日で赤」という段階制を提案していました。v2 では **廃止** します。

廃止根拠（ふくだ君判断）:
- **毎朝、人がオプトインする**ので、連続性をシステムが追っても意味がない
- 連続日数で「翌日も自動的に厳しくする」のではなく、**毎朝の判断時にふくだ君が前日サマリを見て決める**
- 例えば 2 日連続で `ledger_only` が出ていれば、翌朝のオプトイン UI に「前日と前々日に不一致あり」と表示し、ふくだ君が「今日は休む」を選べる。機械が「連続だから自動停止」と決める必要はない

これにより `consecutiveHighDays` カウンタの永続化問題（v1 §8-8 で問題提起した「解除でリセットされてしまう問題」）も自動的に消滅します。**カウンタ自体が存在しない。**

### 2.3 当日停止中のシステム挙動

**3層に分けて整理**:

| 機能 | 停止中の挙動 |
|---|---|
| `vb_refresh`（オッズ更新） | ✅ **続行** |
| ML 予測（predict / predict_closing / generate_bets） | ✅ **続行** |
| Orchestrator（FF出力・承認） | ❌ **停止** |
| UI 表示（/predictions, /odds-race, /auto-purchase） | ✅ **続行**（閲覧可） |
| 自動購入モード | ❌ **stopped** に固定（当日中は変更不可） |
| 新規ベット送信 API | ❌ 全拒否 |
| `/api/bankroll/check` | ✅ 続行 |
| 突合エンジン | ✅ 続行（さらなる 🔴 を検出しても挙動は変わらず、ledger に追記するのみ） |

**設計判断の理由**:

1. **vb_refresh と ML 予測は止めない**: データの連続性が失われると、翌朝オプトインで再開した時に「直近データが古い」状態になる。データは常に新鮮に保ち、購入判断だけ止める。
2. **UI は閲覧可**: ふくだ君が停止中も状況を確認できる。
3. **Orchestrator は完全停止**: 承認・FF出力・キュー処理すべて停止。
4. **当日中の解除手段は無し**: v1 では「手動 stopped は即解除可」「黄/橙はクールダウン後に解除」を許していたが、v2 では当日中の解除は **手段ごと廃止**。一度止まったら、翌朝のオプトイン UI が唯一の再開経路。

> シズネ注: 「気軽に押せる緊急停止 + 当日中の解除不可」の組み合わせがミソです。気軽に押すと午後の予定全部キャンセルになるので、押す時は「今日はもう終わり」の意思を伴います。逆に言うと、押した後に「やっぱり 30 分後から再開したい」という揺らぎを物理的に殺せます。

## 4. 翌日オプトイン UI

**v2.0 で新規追加**。これが本書の中核です。

### 4.1 思想

毎朝、ふくだ君が自動購入を開始するときに **明示的なオプトイン操作** を求める。これにより:

1. 「自動運転が勝手に動いてる」状態が発生しない
2. 前日の状況（不一致・収支・bankroll）を必ず一度見てから当日を始める
3. 「今日は出張で見られないから休もう」「今日は不一致 2 件続いてるから休もう」を **人が判断できる**

Themis 原則との整合: オプトイン UI に表示するのは「**確率・オッズ・bankroll の周辺事実**」のみ。「今日は勝てそうですよ」「今日のおすすめレース」みたいな主観的レコメンドは **絶対に出さない**。事実だけ並べて、判断は人。

| **目的** | 「最近、稼げてるのか溶かしてるのか」を一瞥で判断材料にする。突合状況（②）とは別の収支軸 |

| **ROI 計算式** | `payout / invest`（×100 で %）。日次プロット。**ハズレ・的中問わず、その日の総投資と総払戻のみで算出**（途中停止日は当日の確定分のみで計算）|
| **集計 API** | `/api/auto-purchase/optin-context` の追加フィールド `recent_7day_roi[]` として返す（11 章 §5.4 に集計要件を追記） |
| **チャート UI 仕様（軽量）** | 高さ 60px / 幅 280px。SVG sparkline。色: 100% 超 = 緑、80-100% = 黄、80% 未満 = 赤。損益分岐 100% を破線で表示 |
| **Y 軸スケール (v2.2 確定)** | **auto-scale + 100% 必ず含める** ハイブリッド方式。実装規則: ① 直近 7 日 ROI の min/max を取得 → ② `[min(min,100)-10, max(max,100)+10]` を Y 軸範囲とする → ③ 100% の損益分岐線は常に軸内に描画。理由: 完全固定 (0-200%) は通常変動を視認しづらく、完全 free は損益分岐ラインが画面外に出て解釈ミスを誘発する。**100% を必ず含める** ことで「黒字日か赤字日か」が一瞥で確定する |
| **欠損日扱い** | 当該日に `daily_close` が無い（クローズ未済）→ ライン途切れ + 灰色 "?" マーカー。ふくだ君に「クローズ未済の日がある」を視覚的に伝える |
| **数値併記** | 「直近7日 ROI(投資加重): XX.X%」+「累計収支 ±¥X,XXX」を1行で併記。チャートが小さくても数値で判断可能にする |
| **「最近の傾向」表示なし** | 「下降トレンドです」のような主観的レコメンドは出さない。事実だけ並べて判断は人 (Themis 原則) |

```typescript
interface OptinContextV21 extends OptinContext {
  recent_7day_roi: Array<{
    date: string;          // YYYY-MM-DD
    invest: number;        // 円
    payout: number;        // 円
```

### 4.3 「今日は休む」を選んだ場合 + モード昇格 UI（v2.1）

| 項目 | 挙動 |
|---|---|
| `mode` | `manual` のまま（変更なし） |
| ledger イベント | `MANUAL_OPT_OUT_DAY` を発火 |

| 翌日 | 翌朝同じオプトイン UI が出る |

#### 4.3.1 モード昇格 UI（v2.1 新設）

「開始する」を押した直後は §4.4 のとおり必ず `mode=manual` で始まる。そこから `semi` / `auto` へ上げる遷移は、ダッシュボード上で **以下の最低限の摩擦を経由する**:

```
[manual] ← 開始直後・デフォルト
   │ ボタン: [semi に上げる]  (1クリック目)
   ▼
[確認ダイアログ]
   「semi モードに上げると、Orchestrator がオッズ取得後の
    買い目を 1 レース毎に承認待ちキューに積みます。
    承認ボタンを押せば即購入されます。よろしいですか？」
   [キャンセル] / [semi に上げる]  (2クリック目)
   │
   ▼
[semi] ← 半自動
   │ ボタン: [auto に上げる]  (1クリック目)
   ▼
[確認ダイアログ]
   「auto モードに上げると、人手承認なしで自動投票されます。
    緊急停止以外で止める手段は当日中ありません。
    本当に上げますか？」
   [キャンセル] / [auto に上げる]  (2クリック目)
   │
   ▼
[auto] ← 全自動
```

遷移ルール:

| 遷移 | クリック数 | 確認ダイアログ | ledger イベント |
|---|---|---|---|
| `manual` → `semi` | 2 | 必須 | `MODE_PROMOTED_TO_SEMI` |
| `semi` → `auto` | 2 | 必須（auto はリスク文言を強めに） | `MODE_PROMOTED_TO_AUTO` |
| `auto` → `semi` | 1 (+ ソフト警告 §4.3.1-bis) | 承認待ちキュー > 0 件の時のみ 1 行警告 | `MODE_DEMOTED_TO_SEMI` |
| `semi` → `manual` | 1 (+ ソフト警告 §4.3.1-bis) | 承認待ちキュー > 0 件の時のみ 1 行警告 | `MODE_DEMOTED_TO_MANUAL` |
| 任意モード → `stopped` (緊急停止) | 1 + 確認 | §5.2 のダイアログ | `KILL_SWITCH_TRIGGERED` |

> シズネ注: **降格は即時、昇格は確認 2 クリック**。これが鉄則。降格を渋らせると緊急時に間に合いません。昇格は摩擦を入れて「無意識に auto まで上げた」を防ぎます。確認ダイアログのリスク文言は必ず「人手承認なし」「当日中は止める手段が緊急停止のみ」と明示すること。

#### 4.3.1-bis 降格時のソフト警告（v2.2 確定）

降格自体は即時 (1 クリック) を維持するが、**承認待ちキューに件数が残っている場合のみ** 1 行のソフト警告を挟む。

| 項目 | 内容 |
|---|---|
| **発火条件** | `auto` → `semi` または `semi` → `manual` への降格時、`approval_queue.length > 0` |
| **ダイアログ文言** | 「承認待ちキューに N 件積まれます。よろしいですか？」（N は実件数）|
| **ボタン** | `[キャンセル]` / `[降格する]`（既定フォーカスは `[キャンセル]` 側）|
| **キュー 0 件時** | ダイアログは出さず即時降格（緊急時に間に合わせるため）|
| **ledger イベント** | `MODE_DEMOTED_TO_*` の payload に `approval_queue_size_at_demote: N` を追加 |
| **設計意図** | 降格自体は素早く可能を維持しつつ、「降格直後にキューが宙ぶらりんで放置」を視覚化。誤操作で `auto` から `manual` まで一気に下げて承認待ちが消えたと勘違いするリスクを摩擦で吸収 |

> シズネ注: §4.3.1 本文の「降格は即時 1 クリック」原則は **キュー空時** に限定して維持。キューが残ってる時に "サイレント即時降格" すると、ふくだ君が「あれ、買い目どこ行った？」となる事故が起きる（買い目自体は残るが、承認窓が降格で消える可能性がある — 仕様詳細は §13 突合と要連携）。1 行の警告で十分。理由入力は不要、`[キャンセル]/[降格する]` の 2 択で 0.5 秒の摩擦を与えるだけ。

### 4.4 「開始する」を選んだ場合（v2.1: manual 始動固定）

| 項目 | 挙動 |

## 8. シズネからの注意点（v2.0 版）

1. **「翌朝になったら勝手に再開」を絶対に作らないこと**。これが v2.0 の最重要原則。`mode` を `auto`/`semi` に自動的に戻すコードパスを実装に紛れ込ませないこと。レビュー時に必ずこの点を確認してください。

2. **時刻はサーバ時刻基準**。クライアントの時計改ざんで「当日終了」を早めない。`eod_at` は必ずサーバが計算。

3. **発動条件の変更には監査が必要**。`dd_tiers.yellow_pct: -20` を `-25` に緩めるような変更は、`bankroll/config.json` の変更履歴（[10_BANKROLL_CONTROL.md](./10_BANKROLL_CONTROL.md) §6）に記録され、月次レビューで必ず追跡。

4. **bankroll < 10% は最も重い発動**。これは DD と独立に判定。peak からのドローダウンは小さくても、bankroll が初期 10% を割ったら停止。「ジワジワ溶ける」パターンの最終防衛ラインです。

5. **手動 stopped を「気軽に押せる」状態にする**。気軽に押せないと、いざという時に押せません。ヘッダ常駐ボタン + ホットキー（Ctrl+Shift+S とか）を検討してもいいくらいです。**ただし v2.0 では「押すと当日中は再開できない」のでもう一段重みがあります**。確認ダイアログでその意味を一行明示すること。

6. **当日停止中の精神的影響**: ふくだ君、当日停止中は「自分が悪い」と感じる可能性があります。違います。**システムが「今日はここまで」を担保してるだけ** で、ふくだ君を責めるためのものではない。UI のメッセージング（「今日はここまで。明朝に判断しましょう」のような）は、シズネがそういうトーンで書きます。

7. **「人が毎朝判断する負荷」が新しいリスク**: v2.0 の最大の懸念点。毎朝ボタンを押すだけになって形骸化したら、v1 の「自動復帰」と実質変わりません。これを防ぐために:
   - **オプトイン UI に必ず前日の不一致と直近3開催日のサマリを表示**（§4.2）し、見ないと押せない構造にする
   - 「開始する」「休む」の2択を **3秒以上画面を見ないとボタン活性化しない** などの UI 工夫を Step 2 で検討
   - 月次レビューで「ack せず即押し」率を計測し、形骸化兆候があればシズネが警告

8. **「人が毎朝必ず操作する」前提が崩れる日のケア**: 出張・体調不良・忘れ等で朝オプトイン操作ができない日は **自動的に「休む」扱い**（mode が `manual` のままで購入されない）。これは設計のフェイルセーフであり、ふくだ君がオプトインを忘れた日はバンクロールが減らないだけで損害なし。「忘れると勝手に動く」より「忘れると動かない」が圧倒的に安全。

9. **v1 で書いた「シズネからの注意点 §8-7（手動操作中も突合 kill switch を一段甘くしない）」「§8-8（カウンタ永続化）」は v2.0 で論点ごと消滅**。手動操作中の例外も、カウンタも、そもそも存在しないので考慮不要。

10. **v1 廃止の理由をふくだ君に問われたら**: 「自動復帰を作ると、ふくだ君が気付かない間に資金が動くリスクが残ります。毎朝の人間判断レイヤを残すほうが、機械の連続日数判定より信頼できる」と説明してください。これは Themis 原則の拡張（確率・オッズ・bankroll の3変数の上に「今日動かす意思」を置く）でもあります。

11. **「最終レース終了 + 30分」動的 eod_at の実装注意（v2.1 新設）**: 4 点を必ず守ること:
   - **(i) レーススケジュール取得経路**: 第一優先 `data3/races/{YYYY}/{MM}/{DD}/race_*.json` の `post_time` フィールド（既存ビルダー出力）。第二優先 MySQL `races` テーブル。両方失敗時は §8-11-iv のフォールバックへ。**朝のオプトイン UI 表示時点で eod_at は決定済**にしておくこと（kill switch 発動時に毎回計算するのではなく、当日開始時に一度算出して config に書き込む方針）
   - **(ii) タイムゾーン固定**: 必ず **JST (+09:00)** で計算・記録。サーバが UTC で動いていても、`post_time` は JST 文字列で持ち、JST のまま 33 分加算する。`new Date()` の暗黙変換を絶対に挟まない（夏時間の罠は日本では関係ないが、サーバロケールの罠は実在する）
   - **(iii) 最終レース取消・順延フォールバック**: その日の予定最終レースが取消になった場合 → **「取消前に確定していた予定最終レースの post_time + 33 分」を使う**（取消後の繰り上がりは追わない）。順延（翌日扱い）の場合は当日 23:59:59 JST にフォールバック。`eod_at_source` フィールドに `"fallback_cancelled_lastrace"` 等の理由を記録
   - **(iv) スケジュール取得失敗時のフェイルセーフ**: race_*.json / DB ともに取得失敗 or 当日が無開催日と判定されたら、**必ず `23:59:59 JST` にフォールバック**（旧 v2.0 仕様と同じ）。`eod_at_source: "fallback_no_schedule"` を記録し、UI 上にも「スケジュール取得失敗のためデフォルト 23:59 を使用」と警告表示。沈黙 fallback は絶対に避ける。
   - **追加考慮**: ナイター開催（園田・大井・川崎等の地方は対象外だが、JRA でも開催区分があれば）の終了時刻が 21:00 を超える場合、33 分加算で 21:33 になる。23:59:59 を超えるケースは現状の JRA 開催では発生しないが、コード上は「`eod_at` が翌日 00:00 を跨ぐ場合は当日 23:59:59 にクランプ」を入れておくこと（将来の予防線）。
