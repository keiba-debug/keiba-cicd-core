### 物理フォーマット（`my_marks.py` / `target-mark-reader.ts`）
- ファイル名 `UM{YY}{KAI}{場漢字}.DAT`（例 `UM261東.DAT`）。各 markSet は別フォルダ（1=直下、2-8=`UmaMark{N}`）。
- 1レコード=44byte：`[0:2]`未使用 / `[2:6]`**レース印(4byte SJIS)** / `[6:42]`馬印(18頭×2byte) / `[42:44]`改行。
- `record_index = (day-1)*12 + (race-1)`、馬印 offset = `record*44 + 6 + (uma-1)*2`。
- **馬印（各馬の序列印）とレース印（レース全体の印）は同じレコード内で別領域**。馬印=`[6:42]`、レース印=`[2:6]`。

## 3. スロット予算の意思決定（⑤⑥③の前提）

**現状 8枠：1=手動 / 2=VB / 3=ARd / 4=IDM / 5=パドック / 6=AI評価 / 7=空 / 8=買い軸**

危印を 7 に退避すると**空き 0**。一方これから乗せたいもの：
- **③ 動的AI印**（オッズ変動で買い目に追加された馬の印）→ スロット要否は設計次第（§4）
- **⑤⑥ AI印2 / AI印3**（別視点モデル）→ 各1枠ずつ要 = 最大2枠不足

→ **選択肢**:
1. **TARGET が 9枠以上いけるか検証** → いけるなら UmaMark9/10 を AI印2/3 に。最も素直。
2. **使用頻度の低いスロットを再配分** → VB/ARd/IDM/パドック(2-5)のうち実運用で見てないものを AI印2/3 に転用。ふくだに「どの自動印を実際使ってるか」確認。
3. **1スロットに複数AI視点を多重化しない**（記号空間が混ざり可読性が落ちる）→ 原則1スロット1視点を維持。

**→ ふくだ確認事項**: (i) 危印は 7 退避でよいか (ii) 自動印2-5 で実際に見てるのはどれか (iii) AI印2/3 は別スロットが要る＝TARGET 9枠検証 or 2-5転用 のどちらで進めるか。

## 4. 派生項目への接続メモ

### ③ 買い目に追加された馬への印（オッズ変動後）
- 事実: **自動購入は投票直前（窓内）の DB ライブオッズで買い目を決める**
  （`bettype_scheduler.size_one_race` を `now>=vote_at` の窓内でのみ実行 → `bettype_efficiency._load_combo_odds` → `core.odds_db.get_all_combo_odds`）。
  → 朝のAI印(markSet=6)と、投票時の買い目（オッズ反映後）がズレる。買い足した馬に印がない＝ふくだ不満点。
- **案1: 完全更新** — markSet=6 を投票時オッズで再計算・上書き。
  - 長所: 印が常に最新。短所: 「朝の評価」が消える＝**監査/ML特徴量化の凍結方針(設計書22条件①)と衝突**。◎が時間で動く。
- **案2: 後付け追加印（専用スロット）** — markSet=6（朝の評価）は凍結のまま、別スロットに「投票時点の動的印」を出す。
  - 既存 **markSet=8（買い軸◆◇）がほぼこれ**: ledger 起点で「実際買った軸+相手」を打つ＝オッズ変動で相手が増えれば◇が増える。
  - つまり **③の大部分は markSet=8 で既に達成可能**。「買い目に追加された馬＝◇」で表示される。
  - 残ギャップ: markSet=8 は**確定 ledger 起点**（投票後）。「投票前の最新オッズで“これから買う予定”の馬」をリアルタイムに出すなら、betting_selection.json 起点の別経路 or markSet=8 を選定段階でも書く拡張が要る。
- **→ 推奨たたき台**: 案2（凍結維持＋追加印）。markSet=8 を「買い軸＋オッズ変動で足した相手」の正式スロットに位置づけ、選定段階（投票前）でも書けるよう拡張。案1（6上書き）は監査と相性が悪いので避ける。

### ① 1レース単位の買い方検証ラボ
- ふくだ要望: 「1レース収支1000%以上を10回に1回」「15点三連複が良い時/1点に絞る時」「複勝よりワイド流し」。
- 既存資産: `bettype_efficiency`（軸◎から各券種プランの 的中確率/合成オッズ/期待リターン を一覧化）が**まさに1レース単位の券種比較エンジン**。bankroll シミュ(`/analysis/simulation`)は複数レース横断の複利軌跡で**粒度が違う**。
- **→ 新規「1レース買い方ラボ」**: 1レース選択 → 全7券種×買い方（1点/流し/フォーメーション/点数）の (的中確率, 払戻分布, 回収率期待, **1000%超の確率**, 分散) を並べ、「目標（例: 回収率期待>100% かつ 大当たり確率>10%）でフィルタ」。複勝 vs ワイド流しの直接比較もここ。
- 既存 `bettype_efficiency` を core に、web に専用タブ。bankroll シミュ(全期間)とは別物として新設。

## 6. 相談: ロジック作成後に過去レースへ遡って印を付けられるか（ポラリス使用可）

**結論: 技術的に可能。ただし「表示用」と「成績検証用」で扱いを分ける必要がある。**

### 技術的足場（確認済み）
- `python -m ml.predict --date <過去日>` で過去レースに polaris を当てて推論できる（過去日対応済）。
- `python -m ml.ai_marks.write_ai_marks --date <過去日> --apply` で過去日の DAT に印を書ける
  （`dat_writer` は `race_id → parse_race_id → coord → get_mark_file_path` で**日付からDATパスを算出**するので過去日OK）。
- → 新ロジック（AI印2/3 等）を作ったら **過去日で predict → 印を遡及 apply** の経路で過去レースに付けられる。

### ★用途で2つに分ける（ここが肝）

| 用途 | 可否 | 注意 |
|------|------|------|
| **(A) 表示・レビュー**（過去にこの印を並べて眺める・UI確認） | ✅ 最新 polaris で可 | 「後知恵（当時はこのモデル/印は無かった）」と理解の上で。実運用の判断再現ではない |
| **(B) 成績検証**（この印ロジックは過去に勝てたか＝バックテスト） | ⚠️ 条件付き | **OOS / walk-forward 必須**。最新 polaris を**学習期間内**の過去レースに当てると data leakage（そのレースが訓練データ）で過大評価＝自己欺瞞 |

### leakage を避ける実務（足場は既にある）
- `ml/evaluate_period.py`（保存済み polaris を**指定OOS期間**で評価）と `ml/analyze/selective_oos.py`（OOS集計）で「学習期間外だけ評価」の枠組みがある。
- `--sire-cutoff`（血統統計の未来データカットオフ）= 既にリーク対策が意識されている。遡及付与でも cutoff を当時日付に合わせる。
- **検証目的の遡及は polaris の学習期間より後（OOS）に限定**すれば正当。表示目的なら全期間でも可（後知恵と明記）。

### AI購入軸（markSet=3）の遡及 — オッズ履歴 確認済み（2026-06-06 実DB調査）

購入軸は「**投票時点のライブオッズ**」で決まる（§4 の②事実）。mykeibadb のオッズ保持状況を実査:

| テーブル | 種別 | 保持期間 | 備考 |
|---------|------|---------|------|
| `odds1_tansho/fukusho_jikeiretsu` | **時系列(分単位)** | **2025/01〜** | 1レース約170-181スナップショット＝**投票時点を正確に再現可** |
| `odds1_tansho` / `odds1_fukusho` | 確定(締切) | 2019/02〜 (25,473R) | 単複 |
| `odds2_umaren` / `odds3_wide` | 確定(締切) | 2019/02〜 (25,473R) | 馬連/ワイド |
| `odds4_umatan` / `odds5_sanrenpuku` / `odds6_sanrentan` | 確定(締切) | **2025/03/15〜** (4,344R) | 馬単/三連系 |

→ **2025/03/15 以降は全7券種が揃う**（単複は時系列で投票時点を厳密再現、組合せは確定=締切オッズ≒投票時点。自動投票は締切直前投票なので確定オッズで実害小）。
→ **AI購入軸の遡及は実用期間（2025/03〜）で高精度に可能**。ふくだ案の「未設定でOK」保険は実質不要。
→ 2025/03 より前を遡及するなら単複(+馬連/ワイド)中心に限定（馬単/三連系の確定が無い）。

### 推奨（確定）
- **AI評価(markSet=2) の遡及**: OK（表示・検証とも）。検証は OOS 期間限定 + sire_cutoff。
- **AI購入軸(markSet=3) の遡及**: **OK（2025/03〜全券種・単複は時系列で厳密）**。それ以前は券種限定。
- 「オッズが無い場合 AI評価とイコールになるか」→ **ならない**。購入軸は評価×オッズで券種・相手を選ぶためオッズ必須。オッズ欠如時は **AI評価の◎コピー(markSet2と重複・無意味) ではなく未設定が正**（過去分の割り切り）。今回は実用期間が揃うので未設定にする必要なし。
- 新ロジック検証は必ず walk-forward/OOS で（「過去に勝てた」を最新モデルの後知恵で水増ししない）。
