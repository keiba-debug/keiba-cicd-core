## TL;DR (結論)

**条件付き GO**。 設計の方向性は正しく、 Session 127-129 で確立した「人手最終確認は暗証番号」 という新しい人手ゲートを構造化する書面としては筋が良い。 ただし**「先に直せ」 が 4 件**、 並行で詰めるべき改善要望が 7 件。 メインで進行中の `launcher.py` 骨格は **4 件のうち少なくとも A/B/D を反映した状態でないとマージしない**ことを推奨。

**最大の懸念は 3 点**:

1. **「暗証番号入力 = 最後の人手」 が実態として骨抜きになる構造** — 投票総額や対象レース群の事前承認ゲートを別に置かないと、 ふくだが暗証番号を入力する時点では「今日何にいくら張るのか」 を確認していない (selective_vote.bat は暗証番号入力後に走るため)。 シズネ原則「お金経路の最終認証は人」 の「認証」 が IPAT の認証であって、 「投票内容の認証」 ではない。
2. **auto_dismiss_dialogs の whitelist 管理が未設計** — §6.3 で「title regex / button title の whitelist 必須」 と書いてあるだけで、 どこに置いてどう更新するか、 誰がレビューするかの SOP がない。 TARGET アップデート時にダイアログが増えると「とりあえず OK で進める」 運用に流れるリスク。
3. **IPAT セッション切れ検知が「投票ダイアログが変化したら検知」 という事後型** — §5 の発想は正しいが、 検知できなかった場合に「投票完了したと思って selective_vote.bat が走り出してしまう」 経路を防ぐ pre-flight チェックがない。

これらを潰せば Phase 4-A から実装 GO。 **暗証番号自動入力範囲外** の宣言 (§1.4) は強く支持する、 ただし設計レベルで「将来も範囲外であり続ける」 ための構造的防御策を追加してほしい (詳細は 🔴 C)。

## 「先に直せ」 詳細 (launcher.py 実装着手前)

### 🔴 A. 「最後の人手」 = 暗証番号 だけでは Themis 原則の「お金経路認証」 として不十分

**該当**: §0, §1.4, §3.3 (実装フロー Step 4), §7 (「自動化が人手最終確認を骨抜きにしないか」 → 🟢 評価)

**問題**: 18 では「IPAT 暗証番号入力が唯一の人手」 = それで「お金経路の最終認証」 が満たされていると整理されている。 だが暗証番号入力は **「IPAT という決済システムにログインする認証」** であって、 **「今日この内容で投票することの承認」** ではない。 具体的に:

```
[ふくだ操作の現状フロー]
PC 起動 → 暗証番号入力 → (自動: selective_vote.bat) → 受付番号音声通知
```

このフローだと **ふくだが「今日の投票内容」 を見るタイミングがゼロ**。 selective_bets.json の中身も、 当日の総投票額も、 対象レース数も、 ふくだは確認していない。 受付番号が音声で流れてきた時点で「もう投票済」。

これは 05_RISK_COMPLIANCE.md §2.1 「IPAT 誤入力 → TARGET FAQ: 目視確認 → 人間チェック必須」 と TARGET_IPAT_SOP.md §3 手順 5 「買い目・金額・合計を目視確認 (人間)」 と **完全に矛盾**。 16 §9-1 で「人手最終確認を迂回」 と明記しているのは投票ダイアログ click の話だが、 18 で前段も自動化すると **「ふくだが投票内容を 1 ミリも見ない経路」** が完成してしまう。

そして 12_KILL_SWITCH_COOLDOWN.md §0 で確立した「**毎朝 manual スタート + 2 クリック昇格**」 原則 — これは「今日の投票内容」 を一回見せるための摩擦だった — も、 18 のフローでは存在しないことになる。

**修正方針**:

§3.3 の実装フロー Step 4 (ふくだ暗証番号入力) **直前または直後** に、 「**当日投票プラン承認ゲート**」 を必須化する。 候補:

1. **(推奨) Step 4 直前**: `launcher.wait_ipat_login_ready()` が暗証番号入力フィールド検知後、 selective_bets.json から当日プランを読み込んで音声 + テキストで読み上げる:

   ```
   「本日の投票プランは、 新潟8R 単勝6番、 京都8R ワイド1-3 と 1-10、 単勝1番。
    合計 5 ベット、 4,000 円。 暗証番号を入力すると投票を開始します。」
   ```

   ふくだは「これを聞いて違和感がなければ暗証番号を入力する」 = 暗証番号入力が「内容承認」 の意思表示も兼ねる。 (= 暗証番号入力 = 二重認証: ① IPAT ログイン ② 当日プラン承認)

2. **(代替) Step 4.5 として CLI 確認**: 暗証番号入力後、 selective_vote.bat 実行直前に CLI で `[当日 4,000円 / 5 ベット投票します。 続行? (y/N)]` を必須化。 ただし「半自動の罠」 で y を即押しする習慣が付くと意味が薄れるので、 (1) の音声読み上げの方が筋。

**実装影響**: `launcher.py` に `confirm_daily_plan(plan_json: Path) -> bool` を追加。 notify.py に `notify_daily_plan_summary()` を追加。 18 §3.2 の API リストに 1 つ追加 + §3.3 フローに Step 3.5 挿入が要る。

**緊急度**: 🔴 **先に直せ**。 launcher.py 骨格に含めるか、 含めないなら別 issue を切る。 「暗証番号入力 = 最後の人手」 という単一文言の下に「実は内容確認していない」 経路が紛れ込むのは、 シズネとしては受け入れられない。

**所要**: 1-2 時間 (notify 文面 + selective_bets 読み込み + launcher API 追加)。

### 🔴 B. auto_dismiss_dialogs の whitelist 管理 SOP が不在

**該当**: §6.3 `auto_dismiss_dialogs()` 汎用関数

**問題**: 設計書は

> max_iterations 回まで反復、 timeout_sec 経過で終了。 click 数を返す。

と書いているが、 §7 リスク観点で「title/button title の whitelist 必須」 と注記しているだけで:

- whitelist をどこに置くか (コード内定数? config? JSON?)
- 誰がメンテするか
- 新しいダイアログを追加する時のレビューフロー
- whitelist 外のダイアログが出たらどうするか (中断? skip? エラー?)

の指針がない。 これは 16_TARGET_AUTOCLICK.md と同じ「実機 inspect で取得 → 確定」 パターンで運用するのが筋だが、 §4.1 の表は「タイトル候補」 にとどまっている。

**最悪シナリオ**: TARGET が将来バージョンアップで「JV-Link 接続障害につき設定を初期化しますか」 みたいなダイアログを出した時、 whitelist にマッチしないと放置されてふくだに音声通知が来る (これは OK)。 でも whitelist がゆるく `OK|はい|閉じる` だけで作られていたら、 **意図しない設定変更ダイアログを OK で進めてしまう** リスクがある。

**修正方針**:

§6.3 を以下のように具体化:

1. **whitelist の場所**: `data3/userdata/target_clicker/launch_dialogs.json` (JSON + JSON Schema 付き)
2. **スキーマ**:
   ```json
   {
     "version": "1.0",
     "verified_at": "2026-05-24T10:00:00+09:00",
     "verified_by": "fukuda",
     "target_version": "TARGET frontier JV Ver6.21 Rev002",
     "dialogs": [
       {
         "title_regex": "^TARGET frontier JV$",
         "body_regex": "^ライセンス情報",
         "button": "OK",
         "category": "license_check",
         "notes": "起動時毎回出る"
       }
     ]
   }
   ```
3. **whitelist 外検知時の挙動**:
   - **デフォルト: abort + 音声通知** (「想定外のダイアログ <タイトル> が出ました。 手動で確認してください」)
   - 自動 skip は **設計レベルで禁止**
4. **追加レビューフロー**: 新ダイアログ追加時は `inspect-launch` (16 §10.3 と同じパターン) で dump → JSON 追記 → `verified_by` 更新 → git commit (= 監査記録)
5. **target_version フィールド**: TARGET アップデート時に whitelist を強制再確認させるための gate。 起動時に現 TARGET 実バージョンと比較、 不一致なら音声警告 + manual モード強制。

**追加実装**: `launcher.py` に `load_dialog_whitelist(path)` + `verify_target_version(actual, expected) -> bool` を新設。 「データ駆動 + バージョン検証」 で「whitelist が暗黙の信頼を持つ」 状態を防ぐ。

**緊急度**: 🔴 **先に直せ**。 これがないと §6.3 は「実装したら何でも OK で進める汎用関数」 になる。

**所要**: 1 時間 (スキーマ定義 + loader + ドキュメント追記)。

### 🔴 C. 「暗証番号自動入力は範囲外」 の構造的防御策が宣言のみ

**該当**: §1.4 + §5.1 「抑止すべきアンチパターン」

**問題**: §1.4 は

> **IPAT 暗証番号の自動入力** ← 意図的に範囲外 (シズネ原則 + 05_RISK_COMPLIANCE.md §「人手最終確認」)

と宣言されているが、 これは **文書レベルの宣言のみ** で、 コード/構造レベルで「将来も範囲外であり続ける」 ための歯止めがない。 ふくだが半年後に「便利だから自動化しよう」 と言い出した時、 あるいは future me が「テスト用に keyring から読む実装くらい良いか」 と書き始めた時、 この宣言だけでは止まらない。

これは 02_INTEGRATION_OPTIONS.md §3 「TARGET FAQ が明示する『最終投票は人手』 を公式に受け入れ、 半自動でリスクを抑える」 と同じ原則だが、 02 では「半自動を採択」 という設計選択でガードしていた。 18 では「半自動の最後の砦」 が暗証番号だけになり、 そこの自動化禁止がドキュメント文言だけ。

**修正方針**: 構造的歯止めを 3 つ重ねる。

1. **コードレベルの defense-in-depth**: `launcher.py` に「`launcher` モジュールには `keyring`, `getpass`, `pywinauto.send_keys` で password 系操作を**書かない**」 という原則を冒頭 docstring + ruff カスタムルール (or 単純な静的 check) で表明。 例:

   ```python
   # launcher.py の冒頭に必ず含める
   """
   ANTI-PATTERN GUARD: This module MUST NOT import keyring, do not use
   getpass.getpass(), and MUST NOT send_keys() into IPAT password fields.
   See: docs/auto-purchase/18 §1.4 / 05 §"人手最終確認"
   Violations are blocked by tests/test_launcher_no_password.py
   """
   ```

2. **テストによる強制**: `tests/test_launcher_no_password.py` を新設。 `launcher.py` のソースを文字列で読み、 禁止 import / 禁止 API 呼び出しを正規表現で検出。 違反したら CI で fail。 (これは 25 行程度の簡単なテストで実装可能)

3. **設計書の対称化**: 18 §1.4 を「範囲外」 から「**永久に範囲外 (構造的禁止)**」 に格上げ、 上記 (1)(2) を明記。 § 5.1 の「抑止すべきアンチパターン」 の暗証番号自動入力項に「テストで強制 (tests/test_launcher_no_password.py)」 を追記。

**緊急度**: 🔴 **先に直せ**。 「文書宣言は変えるのが容易、 コード/テストの方が変えにくい」 という不可逆性の非対称を活用する。 launcher.py に最初の 1 行目から書いておけば、 future me が違反する瞬間にテストが落ちる。

**所要**: 30-40 分 (テスト + docstring + 18 §1.4 と §5.1 追記)。

### 🔴 D. IPAT セッション切れ検知が事後型のみで pre-flight 不在

**該当**: §1.3 Phase 4-C, §5 失敗パターン表「IPAT セッション切れ」

**問題**: 18 のセッション切れ対応は

> 投票ダイアログが「再ログインしてください」 系に変化 → 音声通知 + ふくだに再認証要求

という **事後検知型のみ**。 これだと:

- selective_vote.bat が走り出した後、 1 件目の投票ダイアログで初めて気付く
- 1 件目はセッション切れで失敗、 ふくだが再認証、 でも 2 件目以降は再開されない (selective_vote.bat の状態管理は無い前提)
- 結果 「投票したつもりだったが全部失敗していた」 経路が成立し、 ledger には「失敗 N 件」 だけ残る

そして重要なのは、 18 §3.3 の実装フロー Step 5 `wait_ipat_main_ready()` で「IPAT メイン画面到達 = 投票準備完了」 と判定するが、 **IPAT メイン画面が表示されていることとセッションが有効なことは別事象**。 セッション切れの状態でもメイン画面の HTML が描画されていることはありうる (キャッシュ表示 or 「セッション切れダイアログがメイン画面の上にモーダルで重なっている」 等)。

**修正方針**:

`wait_ipat_main_ready()` に **pre-flight セッション検証** を追加:

1. **方法 A (推奨)**: IPAT メイン画面の特定要素 (例: 残高表示、 当日投票履歴リンク) が **新規取得** されているかを確認。 ページ内に「最終認証: HH:MM:SS」 等の表示があれば数分以内かチェック。
2. **方法 B**: selective_vote.bat 実行 **直前** にダミー読み取り (例: TARGET から「投票履歴照会」 メニューを軽く叩いて応答するか) → 失敗したら abort + 音声「セッション切れ疑いがあります。 再認証してください」
3. **方法 C**: selective_vote.bat 内部で 1 件目の投票ダイアログが timeout した場合、 **2 件目以降を投げない**ロジック (現状の selective_vote.bat / runner.py の挙動を要確認 → 必要なら fail-fast 化)

加えて、 §1.3 で「Phase 4-C は Session 131+」 となっているが、 **これは 4-A/B より先**にしないと、 4-A/B が「セッション切れ疑い時に止まらない自動化」 になる。 後述の H 参照。

**緊急度**: 🔴 **先に直せ**。 4-A/B 完了 = 来週末 OOS 検証が前提だが、 OOS で初めて気付くと検証時間を半分溶かす。

**所要**: 1-2 時間 (pre-flight 検知ロジック + selective_vote.bat の fail-fast 化検討)。

### 🟡 E. ToS グレー領域の累積評価が「中」 のまま

**該当**: §7 リスク観点表「TARGET ToS との整合 → 🟡」

**問題**: 16_TARGET_AUTOCLICK.md §9-1 で既に **「人手最終確認を迂回」 = 設計意図と乖離** と明記済。 02_INTEGRATION_OPTIONS.md §3 では「TARGET FAQ が明示する『最終投票は人手』 を公式に受け入れる」 とある。 これらと 18 (= 前段も機械化) を組み合わせると、 ToS グレー領域は **累積的に拡大**している。 評価「中」 は 16 単体の評価のまま使い回されている印象。

**シズネとしての観察**:
- TARGET 利用規約は「**自己責任**」 と「**予告なき機能停止あり**」 が中核。 利用規約違反 / 法令違反ではない。
- ただし JRA-VAN / TARGET 提供元から見れば「設計意図と異なる使い方」 = サポート対象外。 アップデートで突然動かなくなる可能性がより高い。
- IPAT (JRA) の規約は別物。 IPAT 直接操作 (Selenium 等) は規約上明確にアウト寄り。 18 は TARGET 内 Win32 API 経由なので IPAT 規約には直接抵触しない (TARGET が IPAT 連動仕様を独自解析している部分を経由しているだけ)。

**修正方針**: §7 表に「**累積評価**」 列を追加し、 16 + 17 + 18 トータルで:

- TARGET 規約: 🟡 → 🟡 (グレーだが範囲内)
- IPAT 規約: 🟢 (TARGET 経由のみ、 直接操作なし)
- JRA-VAN 提供元サポート: 🟡 → 🔴 (設計意図と乖離した自動化、 アップデートで壊れる前提)

そして「**壊れた時に手動で投票するためのフォールバック SOP**」 を 18 末尾 (or TARGET_IPAT_SOP.md) に追記。 これは Session 130 必須ではなく Session 131+ で OK。

### 🟡 I. wait_ipat_login_ready の 60s タイムアウト妥当性

**該当**: §3.2 `wait_ipat_login_ready(timeout_sec: int = 60)`

**問題**: 60 秒は短い。 ふくだが PC 起動して TARGET 立ち上げて IPAT メニュー出してから暗証番号入力するまでに、 メール確認や他作業で 60 秒越える可能性が普通にある。 §5 の「10s ごとリマインド」 で穴埋めしているが、 リマインダ 6 回出して abort される運用は ふくだのストレス源になる。

**修正方針**: タイムアウトを **5 分 (300s)** に延長、 60 秒ごとに 1 回リマインダ。 timeout 後は「再起動するか手動で続行するか」 を音声で問う (= async input)。

**所要**: 5 分の調整。 並行で OK。
