## 1. 本ドキュメントの目的

本ドキュメントは、KeibaCICDの予測モデル精度向上に向けた特徴量エンジニアリング戦略を開発者と共有するためのものです。

別途進行中の独自分析（調教パターン、レースレベル評価、ラップタイプ分類等）をMLモデルの特徴量として統合する方針と、その技術的な実装方向性について議論し、開発者からのフィードバックを得ることを目的としています。

> **ステータス:** 将来計画（Discussion Draft）  
> 現行の実装を前提に、段階的に組み込む想定です。即座の実装は求めていません。

## 2. 現状の特徴量構成と課題

### 2.1 現行モデル（v5.5）の構成

- **JRA-VAN生データ:** 着順、タイム、オッズ、馬体重、枠順、距離、コースID等
- **分類4モデル:** Model A（全馬）、V（複勝）、W（単勝）、WV（単勝+複勝）
- **回帰モデル:** Reg B（着差margin予測）
- **bet-engine:** gap + predicted_marginフィルタで Win ROI 119.9%

### 2.2 根本的な課題

**JRA-VANの生データは誰でもアクセスできる公開情報**です。同じデータをそのまま機械学習モデルに投入するだけでは差別化が困難です。

現在のgap + marginフィルタは有効ですが、「なぜgapが生まれるのか（=なぜオッズが過小評価するのか）」の構造的な説明が不足しています。独自の分析で中間指標を生成し、モデルが市場の歪みを構造的に捕らえられるようにすることが、次の精度向上の核心です。

### 2.3 モデル分割の検討

| 対象 | 優先度 | 理由 | 推奨対応 | 備考 |
|---|---|---|---|---|
| **障害レース** | **高** | 平地とは別競技。飛越能力・落馬リスクが支配的 | モデルから除外（予測対象外） | 全体の約5%。分離で平地モデルのノイズ削減 |
| **芝 vs ダート** | **中** | 重要特徴量の優先順位が異なる。馬場影響、血統適性等 | 実験で効果検証後に判断 | LightGBMは内部分岐可能だが、分割で精度向上の可能性 |

## 3. コアコンセプト: 独自分析 → 数値指標 → 特徴量

### 3.1 アーキテクチャ概要

```
[JRA-VAN生データ] + [独自分析JSON群]
         |
         v
[分析パイプライン群] ── 各分析が独立して中間指標を生成
         |
         +-- 調教分析 (training_summary/analysis.json → 調教偏差スコア)
         |
         +-- レベル分析 (rating_standards.json → レベル落差スコア)
         |
         +-- ラップ分析 (race_trend_index/type_standards.json → タイプ適性スコア)
         |
         +-- 隠れた好走 (ラップ分析 + 着順/上がりデータ → 展開負けスコア)
         |
         v
[中間指標テーブル] (horse_id × race_id ごとに数値スコア)
         |
         +--→ [experiment.py] 学習時: 特徴量として統合
         +--→ [predict.py]    推論時: 同じロジックで生成・統合
```

### 3.2 設計原則

1. 1馬1レースあたり1つの数値に集約する（モデルに入れるには数値化が必須）
2. 絶対値ではなく相対値（その馬にとっての偏差、レース内での相対位置）
3. 学習時と推論時で同一ロジックを使用（データリーク防止）
4. サンプル不足時のフォールバック（ベイズ平滑化で全体平均に引き戻す）

### 4.1 調教パターン分析 → 調教偏差スコア

#### 既存データの現状

`training_summary.json` が馬ごとの今回調教データを保持しています。予想で単独利用実績がある指標です。

```json

{
  "horseName": "アーガイルショア",
  "lapRank": "B+",         // ラップランク (A+〜C-)
  "timeRank": "2",          // タイムランク

  "weekendTime4F": 68.2,   // 土日4Fタイム
  "weekAgoLocation": "コ", // 1週前の調教場所
  "weekAgoLap": "B+",      // 1週前ラップ
  "weekAgoTime4F": 54.2    // 1週前4Fタイム
}
```

`training_analysis.json` にランク別の統計があります:

```json
// by_lapRank: ランク別の勝率/複勝率
"B+": { sample_size: 7689,  win_rate: 0.095, top3_rate: 0.268 }
"C+": { sample_size: 14521, win_rate: 0.064, top3_rate: 0.194 }

// by_acceleration: 加速ラップの有無
"+": { sample_size: 92514, win_rate: 0.078, top3_rate: 0.227 }
"-": { sample_size: 26309, win_rate: 0.059, top3_rate: 0.190 }

// by_intensity: 調教強度

"一杯":   { sample_size: 22080, win_rate: 0.052, top3_rate: 0.173 }
```

#### 特徴量化のポイント

> **重要:** 絶対ランクだけでなく「その馬にとっての偏差」が予測に効く情報です。調教Aランクの馬が常にAなのか、普段Bの馬が今回Aに上がったのかで意味が全く異なります。

| 出力特徴量 | 内容 | データソース |
|---|---|---|
| **training_lap_rank** | lapRankの数値化（A+=7, A=6, … C-=1） | training_summary.json: lapRank |
| **training_time_rank** | timeRankの数値化 | training_summary.json: timeRank |
| **training_deviation** | 馬ごとの過去N走平均ランクからの偏差（正=いつもより仕上げ良い） | 過去のtraining_summaryの蓄積が必要 |
| **trainer_pattern_score** | 調教師のbest_patternとの一致度（lift値を活用） | trainer_patterns.json: best_patterns[].lift |
| **course_change_flag** | 調教コース種別の変化（CW→坂路等） | training_summary.json: weekendLocation, weekAgoLocation |

### 4.2 降格ローテ分析 → レベル落差スコア

#### 既存データの現状

`rating_standards.json` がグレード別のレーティング基準を保持しています。8,991レース、2022-2026年のデータが基盤です。

```json
// rating_standards.json のグレード別基準例
"G1_古馬":  { mean: 67.81, stdev: 3.20, high: 69.41, low: 66.21 }
"G3_古馬":  { mean: 65.11, stdev: 2.91, high: 66.56, low: 63.65 }
"OP":       { mean: 58.20, stdev: 3.55, high: 59.97, low: 56.42 }
"3勝クラス": { mean: 53.89, stdev: 2.19, high: 54.98, low: 52.79 }
"2勝クラス": { mean: 55.31, stdev: 2.53, high: 56.57, low: 54.05 }
"1勝クラス": { mean: 54.26, stdev: 2.45, high: 55.49, low: 53.03 }
"未勝利":   { mean: 50.68, stdev: 2.59, high: 51.97, low: 49.38 }

// competitiveness (混戦度)
"G1_古馬": mean_race_stdev: 2.63 → "非常に混戦"

// venue_stats: 競馬場×芝/ダのレベル差
"東京_芝": { mean: 55.48, stdev: 5.19 }
"小倉_芝": { mean: 53.57, stdev: 4.29 }
```

#### 降格ローテの本質

降格ローテとは、前走のレースレベルが今回より高かった馬が、相対的に有利な状況を指します。レーティング基準を使って、「前走のレースレベル」と「今回のレースレベル」の差分を定量化します。

> **例:** G3古馬（mean 65.11）を5着だった馬が、今回3勝クラス（mean 53.89）に出走。  
> レベル落差 = 65.11 - 53.89 = **11.22**  
> 着順だけ見ると「5着の馬」だが、レベル差を考慮すると実は能力上位。

| 出力特徴量 | 内容 | データソース |
|---|---|---|
| **prev_race_level** | 前走のレースレベル（グレードmean値） | rating_standards.json: by_grade[grade].rating.mean |
| **current_race_level** | 今回のレースレベル（グレードmean値 or 出走メンバーから推定） | rating_standards.json + 出走メンバーデータ |
| **level_drop_score** | prev_race_level - current_race_level（正=今回のレベルが低い=有利） | 上記2指標の差分 |
| **competitiveness_score** | レースの混戦度（race_stdevの相対値） | rating_standards.json: competitiveness_thresholds |
| **venue_level_offset** | 競馬場×芝/ダのレベル偏差（東京芝は小倉芝より高い等） | rating_standards.json: venue_stats |

#### VB（バリューベット）との関連性

オッズは前走着順に強く引きずられる傾向があります。レベルの高いレースで5着だった馬は「5着の馬」として低く評価されがちですが、レベル差を考慮すると実際には能力上位かもしれません。

> **仮説:** level_drop_score が大きい馬ほど、オッズが実力を反映しにくく、VB候補になりやすい。gap との相関が確認できれば、marginフィルタの精度向上に直結する。

### 4.3 ラップタイプ分析 → タイプ適性スコア

#### 既存データの現状

`race_trend_index.json` が各レースのタイプ分類を完了しています。2020年以降の全レースが対象です。

```json
// race_trend_index.json の各レースデータ
{
  "trend":    "front_loaded",  // v1分類: sprint_finish / even_pace / front_loaded / ...
  "trend_v2": "sustained_hp",  // v2分類: sprint / sprint_mild / long_sprint /
                               //         even / sustained_hp / sustained_strong / sustained_doroashi
  "rpci":  53.8,  // Race Pace Continuity Index (低=瞬発, 高=持続)
  "s3":    33.9,  // 前半3F
  "l3":    39.4,  // 後半3F (上がり3F)
  "lap33": -5.5   // l3 - s3 (負=瞬発戦, 正=前傾斜戦)
}
```

`race_type_standards.json` がコース×距離別の基準と分布を保持しています:

```json
// コース別RPCI基準とタイプ判定しきい値
"Chukyo_Dirt_1200m": {
  rpci: { mean: 51.47, stdev: 0.67 },
  thresholds: { sustained: 51.79, instantaneous: 51.13 }
}

// 頭数によるrpci補正
runner_adjustments["Dirt_1200m-"]: {
  "少頭数(〜8)":   { rpci_offset: -1.24 },
  "中頭数(9-13)":  { rpci_offset: -0.59 },
  "多頭数(14〜)":  { rpci_offset: +0.18 }
}

// コース別タイプ出現分布
race_trend_v2_distribution["Chukyo_Dirt_1200m"]: {
  sprint_mild: 24.9%, even: 29.3%, sustained_hp: 21.6%, sustained_strong: 15.4%
}
```

#### 特徴量化のポイント: 二重の活用

- **レース側の特徴量:** コース形態 + 出走メンバーの脚質構成から「今回のレースタイプ」を事前推定。`race_type_standards.json` の thresholds と runner_adjustments を使用。
- **馬側の特徴量:** 各馬の過去走を `race_trend_index.json` でタイプ分類し、タイプ別成績（複勝率偏差等）を集計。今回の予想タイプとの適性マッチスコアを算出。

| 出力特徴量 | 内容 | データソース |
|---|---|---|
| **predicted_race_type** | 今回のレースタイプ予測（v2分類: 7タイプ） | race_type_standards.json: thresholds + runner_adjustments |
| **horse_sprint_score** | 瞬発戦での過去成績スコア | race_trend_index.json で過去走のタイプを分類→集計 |
| **horse_sustained_score** | 持続戦での過去成績スコア | 同上 |
| **type_match_score** | 今回予想タイプと馬の適性のマッチ度 | 上記2指標の組み合わせ |
| **similar_course_score** | 類似コースでの過去成績 | race_type_standards.json: similar_courses |

### 4.4 隠れた好走分析 → 展開負けスコア

#### コンセプト

着順だけでは「負けた馬」でしかないが、内容分析では「展開負け」であり「能力負け」ではないケースを数値化します。

> **典型例:** 瞬発力勝負で後ろからものすごい瞬発力を使ったが届かなかったパターン。上がり最速を使いながら4着以下——この馬の能力評価を見落とさないことが重要です。

#### ラップタイプ分析との連携

`race_trend_index.json` のデータを活用して、「そのレースが瞬発戦だったか」を判定した上で、馬の上がりタイムと着順の乖離を評価します。

```python
# 展開負け検出の概念ロジック
if (
    race_trend_v2 in ['sprint', 'sprint_mild']  # 瞬発戦だった
    and agari_rank <= 2                          # 上がり最速級
    and finish_order >= 4                        # 着順は凡走
    and corner4_position >= 10                   # 4角後方
):
    hidden_performance_score = high  # 展開負け。能力は高い。
```

| 出力特徴量 | 内容 | データソース |
|---|---|---|
| **hidden_perf_score** | 着順以上の能力発揮度（高い=展開負けの可能性） | race_trend_index + 着順/上がり/通過位置データ |
| **agari_rank_vs_finish** | 上がり順位と着順の乖離（大きい=脚を使ったが届かなかった） | レース結果データ |
| **pace_disadvantage_flag** | ペース不利フラグ（瞬発戦で後方待機等） | race_trend_index: trend_v2 + 通過位置 |

#### 市場の歪みとの直結

このスコアが高い馬は、次走で展開が向けば巻き返す確率が高い。そしてオッズは前走の着順に引きずられるため、過小評価されやすい。gap + margin + hidden_performance の3軸フィルタで、より精度の高いVB検出が期待できます。

## 5. 実装ロードマップ

### 5.1 フェーズ構成

| Phase | 内容 | 前提条件 | 既存データ | 依存関係 |
|---|---|---|---|---|
| **Phase 0** | 障害レース除外 | フィルタ追加のみ | 不要 | なし |
| **Phase 1** | 調教偏差スコア | 過去調教データの蓄積仕組み | training_summary / training_analysis / trainer_patterns | なし |
| **Phase 2** | レベル落差スコア（降格ローテ） | グレードマッピング、レースレベル推定ロジック | rating_standards | なし |
| **Phase 3** | ラップタイプ適性 | 馬×タイプ別成績集計 | race_trend_index / race_type_standards | なし |
| **Phase 4** | 展開負けスコア | 上がり順位・通過位置データ | race_trend_index + レース結果データ | Phase 3 |

各Phaseは独立して検証可能です。1指標ずつ追加してバックテストでROI改善を確認し、効果が確認できたものから本番投入します。

### 5.2 検証方法

- **バックテストROI:** 既存のbet-engine + 新特徴量モデルで、ROIが改善するか
- **ECE:** calibration精度が維持または改善されるか
- **特徴量重要度:** LightGBMのfeature importanceで新特徴量が上位に入るか
- **gap相関分析:** 新スコアとgap（VB指標）の相関があるか

## 6. 現行実装との関係

### 6.1 bet_engine.py への影響

新しい特徴量はLightGBMモデルの精度向上を通じて間接的にbet-engineに影響しますが、`bet_engine.py` 自体のフィルタロジック（gap + margin）は変更不要です。将来的に `hidden_performance_score` 等を直接フィルタ条件に追加する可能性はありますが、それはバックテストで効果が確認できてからの判断です。

### 6.2 フロントエンド移行との関係

現在進行中のフロントエンド移行（useMemo → JSON読み込み）は、本ドキュメントの特徴量追加とは独立しています。フロントエンドはサーバー側で生成された recommendations JSON を表示するだけなので、バックエンドの特徴量が増えてもフロントエンドの変更は不要です。
